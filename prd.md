# PRD — OKX Robot 跟单交易机器人

## 背景与目标

自动跟踪指定链上钱包地址的交易行为，在 Base 链上通过 OKX DEX API 复制其买卖操作，主要针对 Virtuals Protocol 平台发行的代币。

---

## 用户故事

> 作为交易者，我提供一个 Base 链钱包地址，机器人自动检测该地址的每笔交易，并用我自己的钱包按比例或固定金额跟单，同时保护我不超过每日亏损上限。

---

## 功能范围

### Phase 1 — 核心跟单（MVP）

**目标：跑通完整跟单链路，固定比例跟单。**

| 功能 | 描述 |
|------|------|
| 地址监控 | 通过 Base 链 WebSocket 实时监听指定钱包的 pending/confirmed 交易 |
| 交易解析 | 识别 DEX swap 交易，提取 token_in / token_out / amount |
| 跟单执行 | 通过 OKX DEX API 执行相同方向的 swap |
| 比例跟单 | 跟单金额 = 钱包空闲 USDC/USDT 余额 × 固定百分比（默认 10%） |
| Dry-run 模式 | 默认开启，只打印预期操作，不发链上交易 |
| 每日亏损上限 | 当日已实现亏损超过阈值时，停止新跟单（默认 $50） |
| 基础日志 | 每笔跟单记录时间、代币、金额、txhash、状态 |

**不在 Phase 1 范围：**
- 自定义跟单金额
- 多地址跟单
- Web UI / Telegram 通知

---

### Phase 2 — 跟单金额自定义

**目标：支持灵活的仓位控制。**

| 功能 | 描述 |
|------|------|
| 百分比模式 | 空闲余额的 N%，支持运行时修改 |
| 固定金额模式 | 每笔固定 $X USDC/USDT，支持运行时修改 |
| 每笔上限 | 单笔跟单金额不超过 $X |
| 配置热更新 | 修改 config 文件后无需重启生效 |

---

### Phase 3 — 多地址 & 过滤

**目标：支持同时跟踪多个地址，过滤低质量信号。**

| 功能 | 描述 |
|------|------|
| 多地址跟单 | 同时监控最多 5 个地址，各自独立配置 |
| 代币白名单 | 只跟单白名单内的代币合约地址 |
| 最小交易额过滤 | 被跟单地址交易额低于阈值时跳过 |
| 地址权重 | 不同地址分配不同跟单比例 |

---

### Phase 4 — 通知 & 监控面板

**目标：可观测性与远程控制。**

| 功能 | 描述 |
|------|------|
| Telegram Bot | 跟单成功/失败推送，每日盈亏汇报 |
| 简单 CLI 面板 | 实时显示监控状态、今日盈亏、持仓 |
| 历史记录查询 | 按日期/代币查询跟单历史 |
| 远程暂停/恢复 | 通过 Telegram 命令控制机器人 |

---

## 核心约束

1. **私钥安全**：私钥只存 `.env`，不进代码，不进日志
2. **Dry-run 优先**：新部署默认 dry-run，手动确认后切换 live
3. **滑点保护**：OKX DEX 调用时设置最大滑点（默认 1%）
4. **Gas 保护**：执行前检查 ETH 余额是否足够支付 gas
5. **幂等性**：同一笔被跟单 txhash 不重复跟单

---

## 数据模型（核心）

```
CopyTrade
  id            INTEGER PK
  source_tx     TEXT UNIQUE   # 被跟单的 txhash
  source_addr   TEXT          # 被跟单地址
  token_in      TEXT
  token_out     TEXT
  amount_in     DECIMAL
  our_tx        TEXT          # 我们发出的 txhash
  status        TEXT          # pending / success / failed / skipped
  pnl           DECIMAL       # 已实现盈亏（USDC）
  created_at    DATETIME
```

---

## 成功指标（Phase 1）

- 从检测到被跟单交易到发出我方交易，延迟 < 3 秒
- Dry-run 模式下零链上操作
- 每日亏损上限触发后，当日不再发出新交易
